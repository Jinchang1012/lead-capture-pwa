# 名片捕捉 PWA — 專案規格文件

> 本文件是唯一規格來源（Single Source of Truth）。
> 任何新功能動工前必讀本文件；完工後必須回來更新對應章節。
> 版本：與 `package.json` 的 `version` 同步維護「版本歷程」章節。

## 1. 專案概述

**場景**：中國半導體設備展、單人顧展、網路極不穩、時間零碎。
**目標**：5 秒內完成一筆名片捕捉（拍照 → 標籤 → 完成），全程單手操作，完全離線可用。

**三大設計原則（任何新功能都不得違反）：**
1. **離線優先** — 展場零網路也要 100% 功能可用；需要網路的步驟只允許發生在展前（安裝、下載模型）
2. **5 秒主流程** — 拍照→標籤→完成的主路徑不得加入任何阻塞步驟（彈窗、必填、等待）
3. **單手極簡** — 核心按鈕在拇指區、夠大（≥64px）、防誤觸

## 3. 儲存層架構（三層，職責嚴格分離）

```
┌─────────────────────────────────────────────┐
│ IndexedDB (Dexie)        ← 業務資料（leads）│
│   - 名片照片 Blob、語音 Blob、標籤答案、備註│
│   - 模型下載斷點（model_dl DB）             │
├─────────────────────────────────────────────┤
│ localStorage             ← 輕量偏好設定     │
│   - 自訂問題組、字體大小、ready 旗標        │
│   - 上限意識：單 key < 100KB                │
├─────────────────────────────────────────────┤
│ Cache Storage            ← 大型靜態資源     │
│   - vosk-model-v1（32MB 模型 tarball）      │
│   - Workbox precache（app 殼）              │
└─────────────────────────────────────────────┘
```

**規則：**
- Blob 一律進 IndexedDB，嚴禁 base64 進 localStorage
- localStorage 的 key 一律帶版本後綴（如 `custom_products_v1`），改格式就 bump 後綴
- 任何 Cache Storage 的 cache name 一律帶版本（`vosk-model-v1`）

## 4. 資料模型（Dexie）

### 4.1 DB：`lead_capture`

#### Schema 版本歷程
| 版本 | stores | 變更 |
|---|---|---|
| v1 | `leads: 'id, createdAt, grade'` | 初版 |
| v2（v0.0.2 上線） | `leads: 'id, createdAt'` | tags → answers 泛化；移除無效的 grade 索引 |

#### lead 紀錄結構（v2 目標）
```js
{
  id: string,                  // crypto.randomUUID()
  createdAt: number,           // Date.now()

  photoBlob: Blob | null,      // 壓縮後 JPEG（1600px / 85%）
  photoMime: string,

  // 問題組答案（v2）：單選 = 長度 1 的陣列；未答 = 不存在該 key 或空陣列
  // 內建問題組 id：'grade'（單選）、'products'（複選）
  answers: { [questionId: string]: string[] },

  followUp: boolean,           // ⭐ 待跟進標記（v0.0.3 加入）

  textNote: string,
  audioBlob: Blob | null,
  audioMime: string | null,
  audioDuration: number | null,
  transcript: string,
  transcriptEdited: boolean
}
```

#### v1 → v2 遷移規則
```
tags.grade = 'A'            →  answers.grade = ['A']
tags.products = ['x','y']   →  answers.products = ['x','y']
（followUp 預設 false）
```
遷移寫在 `db.js` 的 `db.version(2).upgrade()`，**只能往前，不可回頭**。

### 4.2 DB：`model_dl`（模型下載斷點，獨立 DB）
```
partial: 'url' → { url, chunks: Blob[], totalSize, downloaded }
```

## 6. 問題組（Question Groups）規格 — 核心抽象

> 「客戶分級」「產品關注」與未來所有自訂資格問題，**全部統一為問題組**。
> 嚴禁為個別問題寫特例 UI 或特例儲存。

### 6.1 定義結構（存 localStorage `questions_v1`）
```js
{
  id: string,          // 內建：'grade' | 'products'；自訂：'q_' + uuid8
  title: string,       // 顯示名稱，可改
  type: 'single' | 'multi',
  builtin: boolean,    // 內建組不可刪除、不可改 type，但可改 title 與 options
  options: [
    { key: string, label: string, desc?: string }
    // key 內建組固定（'A','liquid_cooling'...）；自訂選項 'o_' + uuid8
  ]
}
```

### 6.2 預設值
```js
[
  { id: 'grade', title: '客戶分級', type: 'single', builtin: true,
    options: [
      { key: 'A', label: 'A 級', desc: '高意願' },
      { key: 'B', label: 'B 級', desc: '潛在需求' },
      { key: 'C', label: 'C 級', desc: '無效/同業' }
    ]},
  { id: 'products', title: '產品關注', type: 'multi', builtin: true,
    options: [
      { key: 'liquid_cooling', label: '廠務液冷' },
      { key: 'advanced_packaging', label: '先進封裝' },
      { key: 'automation', label: '自動化整合' },
      { key: 'other', label: '其他' }
    ]}
]
```

### 6.3 行為規則
- TagPage 依序渲染所有問題組：`single` → 3 欄 grid + emerald 高亮；`multi` → 2 欄 grid + sky 高亮
- 單選再點同選項 = 取消；複選 toggle
- 點擊即寫 DB（`answers[questionId]`），無「儲存」按鈕
- 刪除選項/問題組**不回頭改舊 lead 資料**；顯示端遇到對不到的 key 直接顯示原 key
- 匯出時以**當下**問題組定義轉 label

### 6.4 Store 實作模式（所有 localStorage store 統一這個寫法）
參考 `src/store/questions.js`：模組級 cache + `Set` 訂閱者 + `useXxx()` hook。
**不要**用 Context 做 localStorage 偏好（Context 保留給有生命週期的資源，如 Vosk 模型實例）。

## 7. 檔案結構與職責

```
src/
├── main.jsx                  # 進入點；BrowserRouter basename
├── App.jsx                   # 路由表 + VoskProvider + 全域元件掛載
├── config.js                 # 常數：模型 URL/大小、錄音上限、storage key
├── db/
│   └── db.js                 # Dexie schema + 遷移 + GRADES 常數 + CRUD helpers
├── store/                    # localStorage 偏好（§6.4 模式）
│   └── questions.js          # 問題組 + 字體大小
├── context/
│   └── VoskContext.jsx       # Vosk 模型生命週期（唯一的 Context）
├── hooks/
│   ├── useLeads.js           # Dexie liveQuery 封裝（讀取一律經過這裡）
│   ├── useRecorder.js        # MediaRecorder + Vosk 雙軌
│   ├── useVoskModel.js       # VoskContext 的精簡介面
│   └── useQuota.js           # storage estimate
├── pages/                    # 一頁一檔；頁面間不互相 import
│   ├── HomePage.jsx          # / ：拍照入口、狀態列、統計
│   ├── TagPage.jsx           # /tag/:id ：問題組 + 語音 + 備註
│   ├── ListPage.jsx          # /list ：清單 + 搜尋篩選 + 刪除
│   ├── ExportPage.jsx        # /export ：統計 + 三格式匯出 + 清空
│   └── SettingsPage.jsx      # /settings ：問題組管理 + 字體
├── components/               # 可複用元件；不直接碰 DB
│   ├── TagButton.jsx         # 問題選項按鈕（tagClass/subClass 接字體）
│   ├── RecordButton.jsx      # 按住說話 FAB
│   ├── LongPressDelete.jsx   # 長按 1.5s 刪除
│   ├── UpdatePrompt.jsx      # SW 新版提示
│   └── VersionWatermark.jsx  # 版本浮水印
└── utils/                    # 純函式；不碰 React
    ├── compress.js           # 影像壓縮
    ├── modelDownload.js      # 斷點續傳 + Wake Lock
    ├── quota.js              # storage estimate/persist
    ├── download.js           # Blob 下載 + 檔名/時間格式
    ├── exportCsv.js          # CSV（含 BOM）
    ├── exportMd.js           # Markdown frontmatter
    └── exportZip.js          # ZIP 完整包 / Markdown 包
```

**分層規則：**
- `utils/` 純函式，不 import React、不碰 DB（匯出函式接收 leads 參數）
- `components/` 不直接 import `db.js`；資料經 props 或 hooks
- `pages/` 是唯一可以組合 db + store + components 的地方
- DB 讀取一律走 `hooks/useLeads.js`（liveQuery）；寫入走 `db.js` 的 helper（`patchLead` 等）

## 8. 頁面地圖與導航

```
/            首頁    ── 拍名片(label→input capture) → /tag/:id
├─ /tag/:id  標籤頁  ── ✓完成 → /（replace）
├─ /list     清單    ── 點項目 → /tag/:id；匯出 → /export
├─ /export   匯出
└─ /settings 設定    ── 問題組管理、字體大小
```

- 主流程頁面跳轉一律 `navigate(..., { replace: true })` 防止返回鍵堆疊
- 拍照流程：**先建 placeholder lead → 立即導向 /tag/:id → 背景壓縮 patch photoBlob**（5 秒原則的關鍵，不得改成等壓縮完才跳頁）

## 9. 匯出規格

三種格式，全部「匯出當下」動態讀問題組定義轉 label：

### 9.1 CSV（單檔，UTF-8 BOM）
欄位：`id, createdAt(ISO+08:00), followUp, <每個問題組一欄（;分隔複選）>, textNote, transcript, transcriptEdited, audioDuration`
排序：followUp 在前，再按 createdAt 倒序。

### 9.2 ZIP 完整包
```
/photos/{id}.jpg
/audio/{id}.webm
/data.csv          ← 同 9.1
/data.json         ← 無損原始（answers 用 key 不轉 label），供未來 re-import
```

### 9.3 Markdown 包
```
/md/{YYYYMMDD-HHMMSS}_{gradeKey}_{id8}.md   ← frontmatter + 語音轉寫/文字補充兩區
/photos/... /audio/...                       ← md 內相對路徑引用
```

## 11. 版本策略

- `package.json` 的 `version` 為準；改版時手動 bump 並寫進 commit message 首行
- patch（0.0.X）：bug fix、小調整
- minor（0.X.0）：新功能組
- user 回報問題時用浮水印上的版本號對 git log

### 版本歷程
| 版本 | 內容 |
|---|---|
| v0.0.0 | MVP：拍照+標籤+語音識別+匯出+清單+設定基礎 |
| v0.0.1 | 產品標籤自管 + 字體大小可調 |

## 12. 開發守則（每次動工前自查）

1. **讀本文件**對應章節；改動偏離規格時先更新規格再寫码
2. 同一概念只允許一種實作模式：
   - localStorage 偏好 → §6.4 store 模式
   - DB 讀取 → useLeads liveQuery
   - 確認彈窗 → 主流程零彈窗；破壞性操作用長按或兩步確認
3. 新 localStorage key → 登記 §5；新 Dexie 版本 → 登記 §4
4. 繁體中文註解；變數/函式英文
5. 改動後驗證：preview 無 console error + 主流程（拍→標→完成）可跑通
6. 每版部署後在 §11 版本歷程補一行

## 13. 已知限制 / 技術債

- Vosk 模型 URL 依賴 ccoreilly CDN；模型總大小硬編碼於 config（CORS 不暴露 Content-Range）
- `ScriptProcessorNode` 已 deprecated（vosk-browser 官方模式），堪用
- OneDrive 資料夾下 build 偶發鎖檔（exit code 9）→ 刪 dist 重跑
- 無 OCR（決策：展場用語音補充；展後可批次 LLM Vision 處理照片）
- iOS 未測試（音檔 webm/opus 在 iOS Safari 不支援播放；資料仍完整）
